## 1. Architecture Overview

**Architecture budget:** `mindful`. This is a port, not a build. The static app (`static/`)
is the visual + content oracle; the live `getPage()` payload is the data contract. Every
component, theme module, and motion controller already exists and is correct — the work is
the deliberate transform from plain React props to **SDK field components** driven by live
Edge data, plus the integration surfaces a real head app needs (component-map, nav, editing-safe
rendering, deploy). No new abstraction is introduced that the static app did not already justify;
the only genuinely new layers are the two HTTP route handlers (§5) and the server/client
component-map split (§2), each cited to a concrete requirement below.

**Simplicity contract.** New surfaces and their justification:
- `/api/nav-children` Route Handler — required by ADR-0011 (lazy L2 nav; a Server Action would
  fire `router.refresh()` → flicker). **Survives to production.**
- `/api/layout-debug` Route Handler — required by the port-drift mitigation (ADR-0005): derive
  props from the **live** payload, not static names. **Dev-only; removed at ship (FR-8).**
- `.sitecore/component-map.ts` (Server) + `.sitecore/component-map.client.ts` (Client) split —
  required because 8 of the static components are `'use client'` (§2.3). Not a new
  abstraction — it is the stock Content SDK App-Router shape.
- 100% stock data layer (`client.getPage()`); **no mock infrastructure** (the tenant is real).
  No state store, queue, or cache beyond Edge defaults.

**Two-track shape (Content SDK head app).** CMS track = the operator's tenant (built ONCE, fully,
up front from `sitecore-content-model.md` + the canonical `componentName` table, then published to
Edge — build-flow Phase 5.5, one operator gate before scaffold). Head-app track = this architecture.
The **sync point** is the canonical `componentName` registry (§4.1, ADR-0010): the source of truth
that binds operator renderings ↔ head-app component-map ↔ live `getPage` payload. A mismatch on any
side renders `<MissingComponent>` silently — caught only at the Home verify gate (ADR-0012).

**Scope inheritance.** ADR-0002 deferred all tenant-coupled work to Act 3; ADR-0009 (scaffold),
ADR-0010 (registry), ADR-0011 (nav), ADR-0012 (Home gate), ADR-0005 (boundary), ADR-0004 (theme),
ADR-0006 (presentational forms) are **binding and not re-litigated here**. This blueprint references
them and documents only the integration delta.

### 2.3 Component-map server/client split (NEW DECISION — ADR candidate A)

**Rule:** a component registers in `component-map.client.ts` (Client) iff the static app marks it
`'use client'` OR it relies on a browser-only API (hooks, refs, event handlers, count-up, gooey,
magnetic, `<img>` `onError` fallbacks, form state). Otherwise it registers in `component-map.ts`
(Server) — pure-presentational leaves and containers that only render fields + children. **Verified
by reading the first line of each `static/src/components/<X>/index.tsx`** (`'use client'` present or
absent) and its hook usage.

| Static component | Static `'use client'`? | Boundary | Why |
|------------------|------------------------|----------|-----|
| Hero | YES | Client | gooey `#gooey` filter + magnetic CTA |
| DestinationsGrid | no | Server | container; renders fields + children only |
| DestinationCard *(child)* | YES | Client | `<img>` error-fallback handler |
| ValueProps | no | Server | container; fields + children |
| ValueCard *(child)* | YES | Client | interactive hover/motion |
| ExperienceShowcase | no | Server | container; fields + children |
| ExperienceItem *(child)* | YES | Client | `<img>` error-fallback handler |
| StatsBand | no | Server | container; fields + children |
| Stat *(child)* | YES | Client | client count-up animation |
| Testimonials | no | Server | container; fields + children |
| TestimonialCard *(child)* | no | Server | pure presentational |
| PromoBand | YES | Client | CSS mesh + magnetic CTA |
| NewsletterCTA | YES | Client | form input state (presentational) |
| Marquee | no | Server | **pure CSS** infinite scroll (0 hooks — verified); derives names from Destinations children |
| PageHero | no | Server | pure presentational |
| RichTextSection | no | Server | `<RichText>` only |
| ContactForm | YES | Client | form input state (presentational, ADR-0006) |
| ContactDetails | no | Server | container; fields + children (NET-NEW rendering) |

> **Registry vs map note.** Only the **13 placeable renderings** (§4.1) need a component-map entry.
> Child templates (DestinationCard, ValueCard, ExperienceItem, Stat, TestimonialCard,
> ContactDetailItem) are NOT separately placed renderings — they are rendered by their container
> reading folder-of-children, so a client child (e.g. `DestinationCard`) is imported by its server
> container and does NOT need its own map entry. A Server Component may render a Client Component
> child, so the container stays server-side while its interactive children hydrate.

**Registry-row map assignment (the 13 placeable renderings only):**
- **Client map (4):** `Hero`, `PromoBand`, `NewsletterCTA`, `ContactForm`.
- **Server map (9):** `PageHero`, `ValueProps`, `DestinationsGrid`, `ExperienceShowcase`,
  `StatsBand`, `Testimonials`, `RichTextSection`, `ContactDetails`, `Marquee`.

This keeps the most data-fetch-coupled code (containers reading children) server-side while the four
genuinely-interactive top-level renderings hydrate on the client.

### 2.4 Non-component infrastructure placement (gen-map trap)

`sitecore-tools generate-map` scans `src/components/` and registers **every** file/folder there as a
Sitecore rendering (per `custom_content-sdk-component-naming-traps`). Therefore: theme provider,
reveal controller, motion helpers, the nav client component + its fetch hook, and any types/queries
**must live in `src/lib/` or `src/ui/`**, NOT under `src/components/<X>/`. The ported `static/src/ui/*`
and `static/src/lib/*` map to `sites/aphelion/src/ui/*` and `src/lib/*` unchanged.

### 4.2 Per-component port mapping rule (plain prop → SDK field component)

The mechanical transform, applied per component, deriving prop NAMES **from the live payload** (probed
via `/api/layout-debug`), NOT from static prop names (ADR-0005 drift trap #1):

| Static React | Ported SDK field component | Field type (`.d.ts`, §4.3) |
|--------------|----------------------------|----------------------------|
| `{title}` / `{props.label}` | `<Text field={fields.Title} />` | `TextField { value?: string \| number }` |
| inner-HTML body | `<RichText field={fields.Body} />` | `RichTextField { value?: string }` |
| `<img src={x.src} alt={x.alt}>` | `<Image field={fields.HeroImage} />` | `ImageField { value?: { src?: string, ... } }` |
| `<a href={cta.href}>{cta.label}</a>` | `<Link field={fields.PrimaryCta} />` | `LinkField { value: { href?, text?, target?, ... } }` |
| `items.map(child => ...)` (container) | read folder-of-children → map per-child field components | `ComponentRendering` children |

**Defensive rule:** optional-chain to the deepest leaf (`fields?.HeroImage?.value?.src`) — empty
datasource fields deliver as present-but-empty (`{ value: {} }` / `{ value: "" }`), not `undefined`
(resolver-patterns empty-state trap). Containers with zero children render an empty section, not a crash.

**Marquee derivation (FR-5):** `Marquee` has no datasource. At render it reads the published
`Destinations` folder's `DestinationCard` children, takes each child's `Name` field in Sitecore source
order, no truncation, and renders the CSS scroll strip. The developer probes the Home layout to
determine how the Destinations children reach the Marquee placement (see OQ-B).

## 5. API Design

Two App-Router Route Handlers. No other backend; the data layer is 100% stock `client.getPage()`.

### 5.1 `/api/nav-children` — production nav infrastructure (ADR-0011)

- **Path:** `src/app/api/nav-children/route.ts` (App-Router Route Handler — **NOT a Server Action**;
  a Server Action auto-fires `router.refresh()` → wipes client hover state → flicker, per
  `custom_nextjs-route-handler-vs-server-action`).
- **Method:** `GET`. **Query:** `?id=<itemId>&site=aphelion&locale=en` (the hovered L1 node).
- **Response (JSON):** `{ children: [{ id, title, url, hasChildren }] }` — L2 nodes, with
  `NavigationTitle` override and `NavigationFilter` respected (per the nav skill).
- **Caller:** the client nav component lazy-loads on hover (≥150ms intent delay), renders the flush
  (no-gap) dropdown. **Survives to production (FR-8).** L1 (root + first level + `hasChildren`) is
  **server-rendered** in the header/footer (no client fetch on initial paint).

### 5.2 `/api/layout-debug` — dev-only probe (removed at ship — FR-8, NFR-5)

- **Path:** `src/app/api/layout-debug/route.ts`. **Method:** `GET`.
- **Query:** `?site=aphelion&locale=en&path=/`. **Response:** the raw `getPage` layout JSON for the
  path — the developer reads it to derive props from the **live** payload (ADR-0005 mitigation), and
  it satisfies the T-scaffold gate `… returns real Home layout`.
- **Lifecycle:** present during dev for per-component probing; **deleted before deploy** (T-deploy
  removes `/api/layout-debug` ONLY — `/api/nav-children` stays). Per `custom_content-sdk-debug-layout-probe`.

### 5.3 Editing / preview (no new endpoint — ADR candidate B)

Editing-safe rendering uses the **stock** App-Router editing surface: the catch-all page calls
`draftMode()` and, when active, fetches via `getPreview` / `getDesignLibraryData` from `searchParams`
instead of `getPage`. Every component must render under that path (resolve in the editor,
optional-chain empty fields, no browser-global at SSR-init). No custom editing route is authored — the
scaffold ships the editing/config/render API routes.

## 8. Security Considerations

- **Secrets:** the Edge contextId → `.env.local` (gitignored, never committed/chat). `.env.example`
  documents var names only. Rule `20-boundaries`: no `.env` editing without explicit ask.
- **Read-only delivery:** the head app only **reads** Edge (`getPage`); no writes, no auth flow, no
  user data store. Forms are presentational (ADR-0006) — no backend, no PII capture this PRD.
- **`/api/layout-debug` exposure:** the probe returns full layout JSON — acceptable in dev, but it is
  a small information-disclosure surface, so it is **removed at ship** (FR-8). Treat as dev-only.
- **`/api/nav-children`:** read-only route-tree data; validate `id`/`site`/`locale` inputs before the
  Edge call; no parameters reach a datastore beyond the Edge query.
- **CSP / headers:** stock scaffold headers; no Marketplace-iframe `frame-ancestors` concern here
  (this is a public site, not an embedded Marketplace app).

## 11. Key decisions (for ADRs)

ADRs 0002, 0004, 0005, 0006, 0009, 0010, 0011, 0012 already exist and are referenced above — **do not
recreate them**. This blueprint surfaces **2 genuinely new** decisions (under the 4-cap; no phase-split):

- **ADR candidate A — Component-map server/client split keyed to the static `'use client'` boundary.**
  Of the 13 placeable renderings, 4 register on `component-map.client.ts` (Hero, PromoBand,
  NewsletterCTA, ContactForm — all `'use client'` in the static app) and 9 on `component-map.ts`
  (Server), with interactive child components (DestinationCard, ValueCard, ExperienceItem, Stat)
  imported as Client children by their Server containers. Rule: a rendering is Client iff its static
  component is `'use client'` / uses a browser API; else Server. Marquee stays Server (pure-CSS,
  0 hooks — verified).

- **ADR candidate B — Editing-safe rendering via the stock App-Router `draftMode()` + `getPreview`/`getDesignLibraryData` path.**
  No custom editing route; the catch-all page branches on `draftMode()` and reads preview data from
  `searchParams`; every ported component must resolve in the component-map under that path and
  optional-chain empty fields so it renders in Pages without crashing (US-5).
