## Problem

We are dogfooding the redesigned Content SDK NEW-site build flow (static-app → port). To exercise it we need a real-feeling marketing site for a cosmos/space brand, **Aphelion**, built the canonical way: a faithful plain-React static app first, then a derived content-model spec the operator can build in Sitecore — no mock infrastructure, no premature head-app scaffolding.

## Goal

Deliver a polished, responsive, dark+light+system-themed **plain-React Next.js static app** (App Router, Tailwind v4, **zero `@sitecore-content-sdk/*`**) for the five Aphelion routes that matches the approved clickdummy under a screenshot-diff gate, plus a mechanically derived `sitecore-content-model.md` runbook. This is Act 1 + Act 2; the head-app port is PRD-001.

## Non-negotiables

- **Zero Sitecore dependencies in the static app.** No `@sitecore-content-sdk/*` packages, no field types, no SDK field components. Plain props only (`title: string`, `imageSrc: string`, `href: string`).
- **One flat prop object per LEAF component**, designed so each scalar prop maps 1:1 to a *future* Sitecore datasource field (this prop set IS the content model). `HeroBanner({ eyebrow, title, subtitle, image, primaryCta })`.
- **Array derivation rule (container/list components).** Array-of-child props (`destinations: Destination[]`, `stats: Stat[]`, etc. in `DestinationsGrid`/`ValueProps`/`ExperienceShowcase`/`StatsBand`/`Testimonials`) map to **child items under a folder datasource** — the canonical SXA Children-resolver pattern (`custom_content-sdk-resolver-patterns`): the container's datasource is a *folder*; each child is an item of the child template carrying the card fields; the container's own scalars (e.g. section heading) become fields on the folder item. NOT multilist, NOT a repeating field.
- **Build mode.** Standard Next.js production build — **NOT** `output: 'export'`/SSG (a static export would misrepresent the head-app SSR/proxy target it ports into). `npm run build` green is a gate.
- **Nav is the one content-model exception.** Header + footer navigation use a static hard-coded tree in the static app and are **excluded from content-model derivation** — they are sourced via the component-level navigation pattern in Act 3 (PRD-001). Do not flag nav as a flat-prop violation.
- **Dark + light + system theme** per `custom_content-sdk-theme-hydration-pattern`: Tailwind v4 `@theme` tokens, FOUC preflight via `public/theme-init.js` (NOT React inline raw-HTML script), `useSyncExternalStore`-based provider, mounted-flag for client-only styling. Theme code is written here and **ports verbatim** to the head app later.
- **No browser globals in render/SSR-init paths** (`typeof window`, `matchMedia`, `IntersectionObserver` belong in `useEffect`) — hydration safety; Playwright is the gate, vitest+build can't catch it.
- **Screenshot-diff fidelity gate** of every route in BOTH themes BEFORE operator review (build from the clickdummy's exact structure, not prose). **Definition:** tight **structural + visual** parity, NOT literal pixel-perfect. Baseline = the operator-selected clickdummy variant rendered in Playwright. Viewports **390 (mobile) + 1440 (desktop)**. Pass = structural parity (same sections, order, layout) + ≤~5% per-route pixel delta per theme.
- **Tasteful motion, not overloaded.** Effects must respect `prefers-reduced-motion`.
- **The content-model spec is derived mechanically** from the static app's component props (one prop → one field, SDK field type chosen by usage).

## In scope / out of scope

- **In scope:** static plain-React app for 5 routes (Home, Destinations, Experiences, About, Contact); full reusable component inventory; theme stack; responsive layouts; component/unit tests + Playwright screenshot-diff; `sitecore-content-model.md` derived spec.
- **Out of scope (→ PRD-001):** Content SDK head-app scaffold, SDK field-component port, live tenant connection, `client.getPage()` wiring, component-level navigation GraphQL, deploy. **Out of scope (deferred entirely):** commerce/cart, i18n/localization, personalization, real forms backend, search.

## Success criteria

- Static app at `products/head-application/aphelion/static/` runs via `npm run dev`, all 5 routes render with no console errors in both themes.
- Screenshot-diff of every route vs the selected clickdummy variant passes the fidelity definition above (structural parity + ≤~5% pixel delta) at 390 + 1440, light AND dark.
- Theme toggle: light / dark / system all work; no FOUC on hard reload; no hydration mismatch warnings.
- Every LEAF component exposes one flat plain-prop object mapping 1:1 to a single Sitecore field; every container/list component follows the array derivation rule (folder-of-child-items); nav is the documented exception; anything else unmappable is flagged net-new in the spec.
- `sitecore-content-model.md` exists: per-rendering table (componentName + datasource fields + types), per-page placeholder composition, content items with verbatim field values, publish targets — authored as a dependency-ordered operator checklist.
- `npm run build`, `npm run lint`, `npx tsc --noEmit` all green.

## Key constraints & assumptions

- App Router, Next.js (scaffold via `create-next-app`, pinned by the scaffold — do NOT hand-write `package.json`, per rule `50-scaffold`).
- Tailwind v4 (`@theme` token model). Hex-literal tokens must NOT be wrapped in `hsl()` (per `hsl_var_token_broken_with_hex_values`).
- Newsletter + Contact are **presentational only** (client-side validation/UI, no submit backend) in v1 — Assumption A1.
- `SITE_COLLECTION_NAME=cosmos`, `SITE_NAME=aphelion` (used by the content-model spec paths; no tenant work in this PRD).
- Imagery: use royalty-free/placeholder cosmos imagery or generated gradients/SVG; do not embed copyrighted brand assets.
