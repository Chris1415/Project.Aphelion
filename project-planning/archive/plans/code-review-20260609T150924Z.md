### MAJOR

#### M-1 — Duplicate non-functional menu trigger in `SiteHeader` (Correctness + A11y)

**File:** `src/components/chrome/SiteHeader/index.tsx:38-48`
**Category:** Correctness, § 4c architecture compliance, Accessibility
**Severity:** Major

**Description:**
`SiteHeader` rendered its own `<button className="menu-toggle" aria-expanded="false">` alongside its primary function. `layout.tsx` renders `<SiteHeader />` (no props) and `<MobileNav />` side by side. The CSS class `.menu-toggle` is `display: none` above 720px and `display: inline-flex` below — so at mobile widths both the `SiteHeader` button and the `MobileNav` trigger button were simultaneously visible.

The `SiteHeader` button had three defects:
1. `onClick={onMenuOpen}` where `onMenuOpen` was never passed from `layout.tsx` → clicking does nothing
2. `aria-expanded="false"` hardcoded — value never changes → WCAG 4.1.2 violation (Name/Role/Value: interactive control state must be programmatically determinable)
3. The comment in `layout.tsx` explicitly states `MobileNav owns the trigger button` — the `SiteHeader` implementation directly contradicted this intent

**Suggested fix:** Remove the dead button from `SiteHeader`; also remove the `onMenuOpen` prop interface, since `MobileNav` is the sole owner of the trigger button.

**Status: FIXED** (see Fixes Applied section)

#### m-2 — `subscribe()` in `theme-provider.tsx`: `window` access outside the guard (Correctness — latent)

**File:** `src/ui/theme-provider.tsx:155-157`
**Category:** Correctness (latent), § 4c architecture compliance
**Severity:** Minor

**Description:**
The `subscribe` function had:
```ts
if (typeof window !== 'undefined') {
  _storeChoice = _readChoice();
  _applyTheme(_storeChoice);
}
// ... closure definitions ...
window.addEventListener('storage', onStorage);   // OUTSIDE the guard
const mql = window.matchMedia('(prefers-color-scheme: dark)');
mql.addEventListener('change', onSystemChange);
```
Lines 155-157 access `window` unconditionally — outside any guard. `useSyncExternalStore` in React 18/19 only calls `subscribe` client-side, so this never throws at runtime. However the guard at lines 134-137 created a false sense that the entire function was SSR-safe. A unit-test environment or any future server-side invocation would throw `ReferenceError: window is not defined`.

**Fix:** Invert the guard to early-return with a no-op teardown when `window` is unavailable, then move all browser-global access into the guarded branch.

**Status: FIXED**

#### m-4 — `ContactForm`: `role="alert" aria-hidden="true"` on always-present placeholder elements (A11y)

**File:** `src/components/ContactForm/index.tsx:160,186,211`
**Category:** Accessibility
**Severity:** Minor

**Description:**
For each field, the component renders a "always-present empty alert placeholder" for `aria-describedby` when no error exists:
```tsx
{!nameError && (
  <p className="field-error sr-only" id="cf-name-error" role="alert" aria-hidden="true" />
)}
```
This pattern is internally contradictory: `role="alert"` declares a live region (AT monitors it for changes); `aria-hidden="true"` tells AT to ignore it. When the error appears (the element is replaced by a visible `role="alert"`), the swap from a hidden element to a visible one produces a non-problematic announcement. However, the empty `role="alert" aria-hidden="true"` element itself violates the ARIA spec — an element cannot simultaneously be both a live region and hidden from the accessibility tree. Some AT implementations may behave unexpectedly.

The simpler and more correct pattern is: keep `aria-describedby` pointing to the error element id always, but only render the error element when there is an error. When the element is absent, `aria-describedby` pointing to a non-existent id is harmless per the spec (AT ignores missing references).

**Suggested fix:** Remove the `{!fieldError && <p ... aria-hidden="true" />}` branches entirely. Keep only the conditional `{fieldError && <p ... role="alert">...</p>}` branches. The `aria-describedby` attribute on the input is still valid when the target element doesn't exist yet.

**Left for operator triage** — not auto-fixed because the existing form tests all pass and the fix changes rendered DOM structure. The change is safe but warrants a deliberate decision. The axe E2E gate should flag this if it registers as a violation at runtime; if the e2e gate passes clean with this pattern, operator can leave it.
