## 1. Architecture Overview

PRD-000 builds **two design-time deliverables, no runtime backend**:

1. **The static app** (`products/head-application/aphelion/static/`) — a standalone Next.js App Router site (TS + Tailwind v4) realizing the approved clickdummy for 5 routes, with **zero `@sitecore-content-sdk/*` dependencies**. Content is hard-coded plain props. This is a reference deliverable, not deployed (ADR-0003).
2. **`sitecore-content-model.md`** — a spec mechanically *derived* from the static app's component prop boundaries (ADR-0005). It is the operator's Act-2 CMS runbook; it describes a *future* Sitecore content model, not a live integration.

The single architectural through-line is **prop-boundary-as-content-model**: every component is shaped so the later Act-3 port (PRD-001) is a mechanical field-component swap, never a restructure. That portability constraint — not any runtime concern — drives the directory layout, the prop-design rules, and the theme stack (which moves verbatim to the head app).

**Inherited decisions (binding, not re-opened here):** ADR-0002 (two-PRD split), ADR-0003 (static-app-first), ADR-0004 (theme stack), ADR-0005 (content-model component boundary), ADR-0006 (presentational forms). This document references them; it does not restate them. The genuinely new decisions are the static-app **stack/scaffold/layout** and the **test + fidelity-gate tooling** (§11).

**SDK-contract gate — N/A for PRD-000.** The static app makes **no live SDK calls** (no `client.*`, no `xmc.*`, no Edge GraphQL). There are therefore **no request/response `.d.ts` shapes to cite** under rule `40-sdk-contracts` in this PRD. The content-model field types in §4 are **design-time targets** (what a future datasource field will be), not observed wire shapes. The SDK-contract verification gate is **deferred in full to PRD-001**, where `client.getPage()` and Edge GraphQL shapes become real and must be cited from `node_modules` `.d.ts`.

## 2. System Components

All components below live inside the single `static/` Next.js app. There is no server tier, no database, no external service.

### 2.1 Route components (App Router pages — server components by default)

| Route | File | Composition |
|-------|------|-------------|
| Home `/` | `src/app/page.tsx` | Hero → ValueProps → DestinationsGrid (preview) → ExperienceShowcase → StatsBand → PromoBand → Testimonials → NewsletterCTA |
| Destinations `/destinations` | `src/app/destinations/page.tsx` | PageHero → DestinationsGrid (full) → NewsletterCTA |
| Experiences `/experiences` | `src/app/experiences/page.tsx` | PageHero → ExperienceShowcase (full) → StatsBand → NewsletterCTA |
| About `/about` | `src/app/about/page.tsx` | PageHero → RichTextSection → ValueProps → Testimonials |
| Contact `/contact` | `src/app/contact/page.tsx` | PageHero → ContactForm (presentational) |

Each page is a thin server component that imports content from a co-located **content module** (`src/content/<route>.ts`) and passes plain props into the reusable components. This keeps the hard-coded content textually adjacent to its future content-model entry, which is what the Act-2 derivation reads.

### 2.2 Reusable component inventory (`src/components/<Name>/`)

One folder per component, each exporting a single typed plain-prop React component plus its `Props` interface. **This is the unit that ports 1:1 to a head-app `src/components/<Name>/` in PRD-001** — so the boundary is the design contract, not a convenience.

- **Leaf (flat-prop) components** (ADR-0005 rule 1): `Hero`, `ValueCard`, `DestinationCard`, `ExperienceItem`, `Stat`, `PromoBand`, `TestimonialCard`, `NewsletterCTA`, `RichTextSection`, `PageHero`, `ContactForm` (presentational, ADR-0006).
- **Container/list components** (ADR-0005 rule 2 — folder-of-children): `ValueProps` (→ `ValueCard[]`), `DestinationsGrid` (→ `DestinationCard[]`), `ExperienceShowcase` (→ `ExperienceItem[]`), `StatsBand` (→ `Stat[]`), `Testimonials` (→ `TestimonialCard[]`). Each accepts an array of child prop objects plus its own scalar(s) (section heading/eyebrow).

### 2.3 Chrome (`src/components/chrome/`)

`SiteHeader` (logo + primary nav + `ThemeToggle`; **static hard-coded nav tree** — ADR-0005 rule 3 nav exception), `SiteFooter` (nav columns, brand blurb, legal, social), `MobileNav` (hamburger drawer ≤640). Mounted in `src/app/layout.tsx` so they wrap every route. Nav data is a local const (`src/content/navigation.ts`), explicitly **excluded** from content-model derivation.

### 2.4 Theme + UI infrastructure (`src/ui/`)

Deliberately **outside `src/components/`** so the head app's `generate-map` scanner never treats theme infra as a Sitecore rendering (per `custom_content-sdk-component-naming-traps`). Contains `theme-provider.tsx`, `theme-toggle.tsx`, and small presentational primitives (Container, Section, Button). Theme code here is written to **port verbatim** into the head app (ADR-0004, NFR-5).

### 2.5 Theme preflight (`public/theme-init.js`)

Synchronous FOUC preflight, served as a real file (not inline). See §8 for the injection mechanics (the forbidden inline-raw-HTML path).

### 2.6 Design-time spec (`sitecore-content-model.md`)

Not code — the Act-2 output, derived mechanically from §2.2/§4. Written during `/implement` tranche T-spec, not at architecture time.

There is **no client→server arrow** anywhere in this diagram. That absence is the architecture.

## 4. Data Model

**No runtime data store.** Content is static TypeScript in `src/content/`. The data model that matters is the **design-time future content model**, derived per ADR-0005. The table below mirrors PRD §10 (the authoritative, path-exact version is produced as `sitecore-content-model.md` in T-spec).

### 4.1 Leaf datasource templates (one prop object → one datasource item)

| Component | Future datasource template | Fields (name : SDK field type — design-time target) |
|-----------|---------------------------|------------------------------------------------------|
| Hero | `Hero` | Eyebrow:Single-Line · Title:Single-Line · Subtitle:Rich Text · Image:Image · PrimaryCta:General Link |
| ValueCard | `Value Prop` | Icon:Single-Line · Title:Single-Line · Body:Rich Text |
| DestinationCard | `Destination` | Name:Single-Line · Tagline:Single-Line · Image:Image · Distance:Single-Line · Detail:Rich Text · Link:General Link |
| ExperienceItem | `Experience` | Title:Single-Line · Summary:Rich Text · Image:Image · Duration:Single-Line · Cta:General Link |
| Stat | `Stat` | Value:Single-Line · Label:Single-Line |
| PromoBand | `Promo` | Heading:Single-Line · Body:Rich Text · Image:Image · Cta:General Link |
| TestimonialCard | `Testimonial` | Quote:Rich Text · Author:Single-Line · Role:Single-Line · Avatar:Image |
| NewsletterCTA | `Newsletter` | Heading:Single-Line · Body:Rich Text · Placeholder:Single-Line · ButtonLabel:Single-Line |
| RichTextSection | `Rich Text Section` | Heading:Single-Line · Body:Rich Text |
| PageHero | `Page Hero` | Title:Single-Line · Subtitle:Rich Text · Image:Image |
| ContactForm *(presentational)* | `Contact Form` | Heading:Single-Line · Intro:Rich Text · NameLabel · EmailLabel · MessageLabel · ButtonLabel (all Single-Line; no submit target) |

### 4.2 Container/list → folder-of-children (ADR-0005 rule 2)

| Container | Child template | Folder-item scalars (or rendering params) |
|-----------|----------------|-------------------------------------------|
| ValueProps | `Value Prop` | Heading |
| DestinationsGrid | `Destination` | Heading · Eyebrow |
| ExperienceShowcase | `Experience` | Heading |
| StatsBand | `Stat` | Heading (optional) |
| Testimonials ("Mission Log") | `Testimonial` | Heading |

The container's datasource is a **folder item**; each card is a child item of the child template; the container's own scalars are fields on the folder (or rendering parameters). **NOT multilist, NOT a repeating field** — this folder-of-children mapping is the load-bearing content-modeling output of the dogfood.

### 4.3 Nav exception (ADR-0005 rule 3)

`SiteHeader` / `SiteFooter` navigation is a hard-coded tree in `src/content/navigation.ts`, **excluded** from derivation. It is the documented exception and must not be flagged as a flat-prop violation. (Sourced via the component-level navigation pattern in Act 3 / PRD-001.)

### 4.4 Derivation rule (the mechanical mapping, design-time only)

For each reusable component, the T-spec derivation applies:

```
leaf component  → one datasource template; for each scalar prop emit one field,
                  field type by prop usage:
                    string rendered as text         → Single-Line Text
                    string rendered as HTML/markup  → Rich Text
                    <img src>                        → Image
                    <a href>                         → General Link
container/list  → one FOLDER template + one CHILD template;
                  array element props → child fields (same rule as leaf);
                  container scalars   → folder fields or rendering parameters
nav             → SKIP (documented exception)
unmappable prop → flag NET-NEW in sitecore-content-model.md for operator decision
```

This is a pure source-of-truth transform from `Props` interfaces + `src/content/*` values. No runtime, no SDK.

## 5. API Design

**None.** PRD-000 has **no backend, no API routes, no data fetching**.

- No `src/app/api/*` route handlers.
- No server actions.
- No `client.getPage()` / Edge GraphQL (that is PRD-001).
- `ContactForm` and `NewsletterCTA` are **presentational only** (ADR-0006): client-side validation + default/focus/error/success/empty UI states, **no network submit**. Submit handlers `preventDefault` and transition local UI state; they never `fetch`.

Any apparent need for an endpoint in this PRD is a scope-creep signal toward Act 3 and must be rejected.

## 6. Infrastructure & Deployment

- **Not deployed.** The static app is a reference deliverable (ADR-0003). "Ship" for PRD-000 = committed + pushed to branch `prd-000`, with the gates green. No Vercel project, no hosting (deploy is PRD-001 / Act 3).
- **Build mode — standard Next.js production build, NOT `output: 'export'`/SSG.** A static export would misrepresent the SSR/proxy head-app target the components port into (PRD §9). `npm run build` (which runs `next build`) green is a **ship gate**; `next build && next start` is also how NFR-1 Lighthouse is measured (soft, non-blocking).
- **Local dev:** `npm run dev`. All 5 routes must render with **zero console errors** in both themes.
- **CI/quality gates (run locally for PRD-000):** `npm run build` · `npm run lint` · `npx tsc --noEmit` · Vitest unit suite · Playwright screenshot-diff + console/hydration check. All green before operator review.
- **Node / package manager:** whatever `create-next-app` pins (rule `50-scaffold` — do **not** hand-write `package.json`; pinned versions drift past the model cutoff).

## 7. Technology Stack

| Layer | Choice | Rationale / constraint |
|-------|--------|------------------------|
| Framework | **Next.js App Router** | Matches the head-app target (App Router) so components + theme port without router translation. |
| Scaffold | **`create-next-app` (App Router, TypeScript, Tailwind v4)** | Canonical greenfield React scaffold; run literally per rule `50-scaffold`. **NOT `create-content-sdk-app`** — that belongs to PRD-001 (the static app must carry zero Sitecore deps, ADR-0003). |
| Language | **TypeScript** | Typed `Props` interfaces ARE the content-model contract (ADR-0005); `tsc --noEmit` is a gate. |
| Styling | **Tailwind v4 (`@theme` token model)** | Per ADR-0004 + theme skill. Config lives in `globals.css` (`@theme`, `@custom-variant dark`) — Tailwind v4 has no `tailwind.config.js`. Hex-literal tokens **must NOT** be wrapped in `hsl()` (use `var(--token)` directly; alpha via `color-mix`/SVG `stopOpacity`) — reference `hsl_var_token_broken_with_hex_values`. |
| Theme runtime | **`useSyncExternalStore` provider + `public/theme-init.js` preflight + mounted-flag** | ADR-0004 + `custom_content-sdk-theme-hydration-pattern`. Ports verbatim to head app (NFR-5). |
| Unit test | **Vitest** (+ React Testing Library) | Prop-render assertions + theme-provider behaviour; fast, no browser. |
| E2E / fidelity | **Playwright** (`toHaveScreenshot`) | Screenshot-diff is the fidelity gate; also the **only** catcher of hydration mismatch / FOUC / browser-global-in-render bugs (vitest + build cannot — NFR-2, PRD risk #1). |
| Imagery | Royalty-free / placeholder / generated (gradients, starfield SVG/canvas) | No copyrighted brand assets (PRD §9). |

**Zero `@sitecore-content-sdk/*` packages** is the defining constraint of the entire stack (ADR-0003). A lint/CI assertion that no such import exists is warranted (see §11 testing decision).

## 8. Security Considerations

This is a static, backendless, undeployed reference app, so the attack surface is minimal. The one real "security tooling" edge is the preflight script.

- **Theme preflight injection — use a real file, never inline raw-HTML.** `public/theme-init.js` is referenced from the root layout `<head>` as `<script src="/theme-init.js"></script>` (parser-blocking, synchronous, no `async`/`defer`). **Do NOT** use React's inline raw-HTML injection prop (the `{ __html: '...' }` form) to inline the preflight — security tooling flags it as a false-positive XSS sink and blocks the write (per the theme skill Hard Rules). This is the specific failure mode the architecture forbids; the `<script src>` form gives identical parser-blocking behaviour without tripping the hook. Lint exceptions: `{/* eslint-disable-next-line @next/next/no-sync-scripts */}` on the tag (the sync-script rule is a generic perf hint that doesn't apply to a 5-line preflight).
- **`suppressHydrationWarning`** sits only on `<html>` (its own attribute mismatch from the preflight-set `.dark` class); it does NOT mask child mismatches — those are handled by the mounted-flag pattern.
- **No secrets, no env-gated credentials, no auth** in PRD-000. (`NEXT_PUBLIC_SHOW_THEME_TOGGLE` may gate toggle visibility but carries no secret.)
- **Presentational forms** never transmit user input anywhere (ADR-0006) — no PII handling, no submit endpoint, so no input-exfiltration surface.
- **Imagery** must be royalty-free/placeholder/generated — a licensing (not security) constraint, enforced at review.

## 9. Risks & Tradeoffs

| Risk / tradeoff | Severity | Mitigation (architectural) |
|-----------------|----------|----------------------------|
| Theme hydration mismatch / FOUC | High | Follow `custom_content-sdk-theme-hydration-pattern` exactly: external preflight, `useSyncExternalStore` (server snapshot = `'system'`), mounted-flag. **Playwright** console/hydration check is the gate (vitest + build cannot catch — NFR-2). |
| A prop that won't map to a single field → painful Act-3 port | High | ADR-0005 boundary enforced at design **and** code review; container arrays use folder-of-children; any net-new flagged in the spec. No computed/nested "convenience" props. |
| Design drift (built from prose, not clickdummy markup) | Med | **Build from the selected clickdummy's exact structure** (`feedback_poc_is_implementation_contract`); Playwright screenshot-diff BEFORE operator review. |
| Browser globals in render/SSR-init path | Med | Lint + review forbid `typeof window`/`matchMedia`/`IntersectionObserver` in render bodies; they live in `useEffect`. Playwright is the catch-all gate. |
| Tailwind v4 hex-token-in-`hsl()` breakage | Med | `var(--token)` directly; alpha via `color-mix`/SVG `stopOpacity` (`hsl_var_token_broken_with_hex_values`). |
| Scope creep toward Act 3 inside PRD-000 | Med | Hard rule: zero `@sitecore-content-sdk/*` import (CI-assertable); no API routes / server actions / data fetching. |
| `ui/` infra accidentally placed under `components/` → becomes a head-app rendering later | Low (here) / High (PRD-001) | Keep theme + primitives in `src/ui/`, never `src/components/` (`custom_content-sdk-component-naming-traps`). Establishing this now avoids a rename in the port. |
| Fidelity gate flakiness (anti-aliasing/font-load jitter) | Med | Stabilize with Playwright `toHaveScreenshot` thresholds (`maxDiffPixelRatio` ≈ 0.05), disable animations during capture, wait for fonts; structural parity + ≤~5% delta is the PRD definition (not pixel-perfect). |

## 11. Key decisions (for ADRs)

ADR-0002…0006 already cover the split, static-first ordering, theme stack, content-model component boundary, and presentational forms — **not re-opened**. Only genuinely new decisions below (2 proposed; well under the density-gate ceiling — the work is already well-scoped, no phase split warranted).

- **ADR-0007 — Static-app stack, scaffold, and portable directory layout.** Scaffold the static app with `create-next-app` (App Router, TypeScript, Tailwind v4) run literally per rule `50-scaffold` — explicitly **not** `create-content-sdk-app` (that is PRD-001). Adopt a directory layout where each reusable component is a self-contained `src/components/<Name>/` folder that ports 1:1 into the head app, hard-coded content lives in `src/content/*`, and **theme/UI infrastructure lives in `src/ui/` (never `src/components/`)** so the future `generate-map` scanner never treats it as a Sitecore rendering. Consequence: the Act-3 port becomes a mechanical move, and a zero-`@sitecore-content-sdk/*` invariant is enforceable.

- **ADR-0008 — Test + fidelity-gate strategy: Vitest for props/theme, Playwright `toHaveScreenshot` for fidelity + hydration.** Two-layer test stack: Vitest (+ RTL) for prop-render and theme-provider behaviour; Playwright for the screenshot-diff fidelity gate (structural + visual, ≤~5% delta via `maxDiffPixelRatio`, 390 + 1440, both themes, all 5 routes) **and** the hydration/console-error check that is otherwise uncatchable. The fidelity baseline is the operator-selected clickdummy variant rendered in Playwright; build from clickdummy markup, not prose. Consequence: the one class of bug vitest+build structurally cannot see (hydration/FOUC/browser-global-in-render) has an owning gate, and design drift is caught before operator review.
