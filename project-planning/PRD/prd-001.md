## 1. Executive Summary

PRD-001 is **Act 3** of the Aphelion Content SDK NEW-site build: take the operator-selected **v5b "Flux"** design — already realized as a standalone static app and a derived content-model spec in PRD-000 — and turn it into a **real Sitecore XM Cloud Content SDK head application** serving **live Edge data**, then **deploy to Vercel**.

This is an **integration + port** job, not a redesign. The static app (`static/`) is the visual and prop-shape oracle; the live `getPage()` payload is the data contract. Each plain-React component is transformed into a Sitecore-driven one (plain props → SDK field components), registered in the component-map, and verified against the live tenant. The header/footer nav — excluded from the content model in PRD-000 — is built with the component-level navigation pattern.

The defining new constraint vs PRD-000: **real-tenant smoke is now a load-bearing gate** (no `<MissingComponent>`, every rendering shows expected content, nav works, editor/preview safe). To de-risk the first real-tenant port, the **Home route is a hard verify gate**: build → publish → scaffold → port → smoke Home *first*; the four inner routes (mechanical repetition of the same components) follow only once Home is green.

## 9. Technical Considerations

- **Build-flow reference:** `custom_content-sdk-website-build-flow` Act-3 track (Phase 1 scaffold → Phase 4 nav → Phase 5 port loop → Phase 6 smoke → Phase 7 deploy). EXISTING-site-identical from scaffold onward.
- **SDK shapes:** `TextField`/`RichTextField`/`ImageField`/`LinkField` and `Field<T>` from `@sitecore-content-sdk/react` `.d.ts` — cite inline; the SDK-contract gate applies this run (unlike PRD-000).
- **Component-map:** App Router `.sitecore/component-map.ts` (Server) + `component-map.client.ts` (Client) — register each rendering's `componentName` exactly; `'use client'` components (gooey Hero, magnetic CTAs, forms, count-up) register on the client map.
- **Port drift trap (ADR-0005):** derive props from the live payload, not static names; re-probe `/api/layout-debug` after wiring.
- **Naming traps:** helpers/types/queries/route-handlers OUTSIDE `src/components/<X>/` (gen-map scans it); avoid generic rendering-parameter names (use `GridLimit`, namespaced).
- **Reads vs mutations:** nav L2 + any client fetch via Route Handler; never a Server Action (auto-`router.refresh()` flicker).
- **Theme port:** verbatim from PRD-000; the `public/theme-init.js`, reveal controller, tokens, fonts move as-is.
- **ADR candidates:** (a) scaffold via baseline-copy + `create-content-sdk-app`; (b) component-map registration strategy (server vs client split); (c) component-level navigation as the nav pattern; (d) Home-route verify-gate tranche; (e) editing-safe rendering approach.

## 10. Data Model Overview

The content model is **already specified** in `project-planning/sitecore-content-model.md` (PRD-000 deliverable), finalized this run with the 3 locked decisions. PRD-001 does not redesign it — it **consumes** it via live Edge. Indicative live-payload contract per component (verified via `/api/layout-debug`):

| Component | Sitecore shape (live) | SDK field components |
|-----------|----------------------|----------------------|
| Hero (leaf) | `Hero` datasource: Eyebrow/Title/Subtitle/Image/PrimaryCta + Meta1–3 | `<Text>` `<RichText>` `<Image>` `<Link>` |
| DestinationsGrid (container) | folder datasource → `Destination` children | per-child field components; `GridLimit` param |
| Marquee | derived from Destinations items (no datasource) | plain text from child names |
| ContactDetails (container) | own rendering → `ContactDetailItem` children | per-child field components |
| (others) | per `sitecore-content-model.md` | matching field components |

The **live `getPage` payload is the binding contract** — the table is the design-time expectation; the port derives props from the actual probed shape.

## 13. Risks and Mitigations

| Risk | Severity | Mitigation |
